## Purpose

In multi-person teams, the `## Team Standards` section of `context.md` is the **shared contract** — every AI agent (Claude, Copilot) on every team member's machine reads the same standards. This ensures consistent code generation across the team regardless of which AI tool or IDE each person uses.

## Steps

### 1. Read Stack
Read `openspec/config.yaml` → extract `x-conductor.stack` fields (language, framework, version, runtime).
If config doesn't exist → note "Not yet initialized — run /sdd-init first".

### 2. Scan Project Config Files
Detect and extract actionable rules from:

| File | What it controls |
|------|-----------------|
| `.editorconfig` | Indentation, charset, line endings |
| `prettier.config.*` / `.prettierrc*` | Code formatting rules |
| `eslint.config.*` / `.eslintrc*` | Linting rules, code quality |
| `tsconfig.json` (strict settings) | TypeScript strictness level |
| `.stylelintrc*` | CSS/SCSS linting |
| `biome.json` | Formatting + linting (Biome) |
| `ruff.toml` / `pyproject.toml [tool.ruff]` | Python linting |
| `clippy.toml` | Rust linting |

Extract only rules that affect code generation (indentation, naming, strictness, banned patterns).

### 3. Scan Team Principles & Lessons
If present, extract actionable guidance from:
- `openspec/principles.md` — non-negotiable project principles
- `openspec/lessons-learned.md` — ecosystem gotchas and design insights

### 4. Scan Custom Skills (project-level only)
Glob for `*/SKILL.md` in project-level skill directories ONLY:
- `{root}/.claude/skills/`, `{root}/.github/skills/`, `{root}/skills/`

**SKIP**: `sdd-*`, `_shared`, `conventions`. Do NOT scan user-level directories (`~/.claude/skills/`).

For each custom skill found: extract name, trigger, and compact rules (5–15 lines).

### 5. Update `openspec/context.md` — Team Standards section
Read the existing `openspec/context.md`. Preserve ALL sections above `## Team Standards` (Stack, Architecture, Key Directories, etc.). MERGE with existing `## Team Standards` section. Preserve any manual annotations (subsections not generated by scanning config files). Only update subsections that correspond to detected config files (Formatting & Linting, Angular, Testing, Pre-commit, Project Config Files). If a subsection exists but was manually added (not matching any scanned config), preserve it unchanged. Update with:

```markdown
---

## Team Standards
_Auto-updated by /conventions on {date}. Re-run to refresh._

### Formatting & Linting
{extracted from project config files — formatting, linting, type strictness}

### Principles
{from openspec/principles.md, or omit subsection if not present}

### Custom Skills
| Trigger | Skill | Description |
|---------|-------|-------------|
{one row per custom non-sdd skill, or omit subsection if none}

### Compact Rules
#### {skill-name}
{5-15 actionable rules per skill}

### Project Config Files
| File | Path | Affects |
|------|------|---------|
{detected config files and what they control}
```

### 6. Persist
- Update `openspec/context.md` (canonical — single file for context + standards).
- File is version-controlled.

## Rules
- Compact rules: 5–15 lines per skill — concise, actionable, no filler
- If no custom skills found: still update Team Standards with formatting/linting rules
- Do NOT scan user-level skill directories — only project-level
- context.md is a TEAM artifact — commit it, review it, keep it updated
- When re-running, MERGE new findings with existing manual additions in Team Standards (don't overwrite)
- PRESERVE all sections above ## Team Standards — only modify from ## Team Standards downward

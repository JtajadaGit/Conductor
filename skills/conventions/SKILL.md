## Purpose

In multi-person teams, `conventions.md` is the **shared contract** — every AI agent (Claude, Copilot, Cursor) on every team member's machine reads the same conventions. This ensures consistent code generation across the team regardless of which AI tool or IDE each person uses.

## Steps

### 1. Read Stack
Read `openspec/config.yaml` → extract `context` field for stack info.
If config doesn't exist → note "Not yet initialized — run /sdd-init first".

### 2. Scan Project Config Files
Detect and extract actionable rules from:

| File | What it controls |
|------|-----------------|
| `.editorconfig` | Indentation, charset, line endings |
| `prettier.config.*` / `.prettierrc*` | Code formatting rules |
| `eslint.config.*` / `.eslintrc*` | Linting rules, code quality |
| `tsconfig.json` (strict settings) | TypeScript strictness level |
| `.stylelintrc*` | CSS/SCSS linting |
| `biome.json` | Formatting + linting (Biome) |
| `ruff.toml` / `pyproject.toml [tool.ruff]` | Python linting |
| `clippy.toml` | Rust linting |

Extract only rules that affect code generation (indentation, naming, strictness, banned patterns).

### 3. Scan Team Principles & Lessons
If present, extract actionable guidance from:
- `openspec/principles.md` — non-negotiable project principles
- `openspec/lessons-learned.md` — ecosystem gotchas and design insights

### 4. Scan Custom Skills (project-level only)
Glob for `*/SKILL.md` in project-level skill directories ONLY:
- `{root}/.claude/skills/`, `{root}/.github/skills/`, `{root}/skills/`

**SKIP**: `sdd-*`, `_shared`, `conventions`. Do NOT scan user-level directories (`~/.claude/skills/`).

For each custom skill found: extract name, trigger, and compact rules (5–15 lines).

### 5. Write `openspec/conventions.md`
```markdown
# Project Conventions
_Auto-generated by /conventions. Re-run to update._
_Shared across all team members and AI agents. Commit this file._

## Stack
{from openspec/config.yaml context field, or "Not yet initialized — run /sdd-init"}

## Team Standards
{extracted from project config files — formatting, linting, type strictness}

## Principles
{from openspec/principles.md, or omit section if not present}

## Custom Skills
| Trigger | Skill | Description |
|---------|-------|-------------|
{one row per custom non-sdd skill, or omit section if none}

## Compact Rules
### {skill-name}
{5-15 actionable rules per skill}

## Project Config Files
| File | Path | Affects |
|------|------|---------|
{detected config files and what they control}
```

### 6. Copilot adapter (optional)
If `.github/copilot-instructions.md` exists (Copilot detected), ALSO write to `.github/instructions/conventions.instructions.md` with `applyTo: "**/*"` frontmatter.

### 7. Persist
- Write `openspec/conventions.md` (canonical).
- Optionally write `.github/instructions/conventions.instructions.md` (Copilot adapter).
- Both files are version-controlled.

## Rules
- Compact rules: 5–15 lines per skill — concise, actionable, no filler
- If no custom skills found: still generate conventions.md with Stack + Team Standards
- Do NOT scan user-level skill directories — only project-level
- conventions.md is a TEAM artifact — commit it, review it, keep it updated
- When re-running, MERGE new findings with existing manual additions (don't overwrite)
